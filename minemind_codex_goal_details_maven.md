# MineMind Codex Goal 详细改造方案

> 用法：把本 Markdown 文档与 Steve 项目源码一起提供给 Codex。  
> Codex 的 Goal 输入框只放精简 Prompt，本文件作为详细需求文档/实施说明。

---

## 0. 项目背景

基于开源项目 **Steve** 改造。Steve 当前是一个 Minecraft Forge Mod 中的 AI Agent：玩家通过 GUI、命令或自然语言任务驱动 Steve 行动，Steve 通过 LLM、任务规划、动作执行系统完成挖矿、建造、跟随、战斗等任务。

MineMind 的目标不是重写 Steve，也不是做一个更强的命令型助手，而是把 Steve 改造成：

> 一个 Minecraft AI 社群模拟系统。  
> 多个 AI 智能体可以在世界中自主观察、自主探索、自主生存、长期记忆、聊天对话、接受玩家引导、彼此交流、分工协作，并逐渐形成自己的社群。

---

## 1. 总体原则

1. 不要推倒重写。
2. 不要另起炉灶。
3. 不要删除 Steve 原有 GUI/命令/自然语言任务能力。
4. 必须基于 Steve 当前真实代码结构渐进式改造。
5. 优先复用 Steve 现有能力：实体、LLM、Action、Memory、WorldKnowledge、多 Agent 协作等。
6. 每个阶段只做一个清晰目标。
7. 每个阶段结束后必须运行 `mvn clean package`。
8. 每个阶段给出中文 commit message。
9. 新功能默认应可配置关闭。
10. 所有自主行为必须可暂停、可停止、可恢复。
11. LLM 请求不得阻塞 Minecraft 主线程。
12. LLM 输出必须校验后才能执行。
13. LLM 失败时必须 fallback 到规则系统。
14. 不允许让 LLM 直接执行任意代码。
15. 不允许硬编码 API Key。
16. 不允许频繁刷屏聊天框。
17. 不允许高频写文件导致卡顿。
18. 多智能体任务分配要避免重复和冲突。

## 2. 最终能力目标

### 2.1 原 Steve 能力保留

- 原 Steve 的生成方式继续可用。
- 原 Steve 的 GUI 输入任务继续可用。
- 原 Steve 的自然语言任务执行继续可用。
- 原有 Action 执行体系继续可用。
- 原有多 Steve 协作能力尽量保留。

### 2.2 单智能体自主模式

- Steve 可以在没有玩家命令时自己行动。
- 它能观察周围环境。
- 它能根据血量、饥饿、时间、附近实体、附近资源、危险等级等信息产生目标。
- 它能在安全时探索，在危险时保命。
- 它能记录自己去过哪里、发现了什么、失败过什么。
- 它能解释自己为什么这么做。

### 2.3 聊天引导模式

- 玩家可以通过 Minecraft 聊天框和 Steve 对话。
- 玩家可以询问“你在干什么”“你为什么这么做”“接下来准备干什么”。
- 玩家可以引导它“不要去东边”“记住这里是基地”“今天优先找村庄”“停止当前行动”“继续自主探索”。
- 聊天引导应写入记忆，并影响后续自主决策。
- 聊天不是强命令系统，而是自然引导系统。

### 2.4 多智能体社群模式

- 可以生成多个智能体。
- 每个智能体有名字、角色、状态、当前目标和记忆。
- 多个智能体可以分工，例如探索者、采集者、建造者、守卫、矿工、农民。
- 社群有公共任务看板。
- 社群有共享记忆。
- 社群有基地概念。
- 社群成员可以交流发现、危险、任务状态。
- 社群可以在没有玩家命令时运行至少一个 Minecraft 日夜循环。

### 2.5 DeepSeek 支持

- 在 Steve 现有 LLM Provider 基础上增加 DeepSeek。
- DeepSeek 可用于命令模式。
- DeepSeek 可用于自主规划模式。
- DeepSeek 可用于低频社群战略规划。
- DeepSeek 不应直接控制底层 Minecraft API。
- DeepSeek 输出必须被校验。

## 2.6 构建系统要求：迁移到 Maven

本项目不要继续使用 Maven（原 Gradle 仅作迁移参考）/Maven 作为主要构建方式。  
Codex 必须在功能改造前，先把 Steve 项目改造成 Maven 构建。

要求：

1. 新增并维护 `pom.xml` 作为主构建文件。
2. Maven 必须能完成：
   - 编译 Java 源码
   - 处理 Forge/Minecraft 依赖
   - 处理资源文件
   - 打包可安装的 Mod JAR
   - 支持本地开发运行或给出等价 Maven 运行方式
3. 不要依赖 `Maven` 作为验收方式。
4. 如果原项目中仍保留 Maven（原 Gradle 仅作迁移参考） 文件，只能作为历史兼容，不得作为主构建入口。
5. 后续所有阶段统一使用：

```bash
mvn clean package
```

6. 如果 Forge 生态的 Maven 构建存在限制，必须优先调研可行 Maven 插件/方案，并在文档中说明取舍。
7. Maven 迁移完成前，不要开始 MineMind 功能开发。

## 4. 阶段 1：新增 MineMind 模式开关

### 目标

在不影响原功能的前提下，给 Steve 增加 MineMind 自主能力的总开关。

### 任务

1. 在现有配置系统中加入 MineMind 相关配置。
2. 支持启用/关闭自主模式。

6. 支持配置思考间隔、最大规划步数、是否使用 LLM 规划。

   - 开启某个 Steve 的自主模式
   - 关闭某个 Steve 的自主模式
   - 查看某个 Steve 的自主状态

8. 自主模式关闭时，Steve 行为必须和原项目一致。

### 验收

- 可以通过命令打开/关闭自主模式。
- 可以查看状态。
- 原 Steve 命令模式和 GUI 模式不受影响。
- 通过 `mvn clean package` 构建。

## 5. 阶段 2：新增世界观察能力

### 目标

让 Steve 能把自己当前处境整理成结构化世界状态，供规则系统和 LLM 使用。

### 任务

1. 基于现有 Minecraft/Forge API 和 Steve 现有 WorldKnowledge 能力实现世界观察。
2. 观察内容应包括：
   - 位置
   - 血量
   - 饥饿
   - 维度
   - 生物群系
   - 时间/昼夜
   - 附近玩家
   - 附近友好生物
   - 附近敌对生物
   - 附近方块摘要
   - 附近资源摘要
   - 背包摘要
   - 当前动作
   - 当前目标
   - 最近失败
   - 危险等级
   - 危险原因
3. 扫描范围要有限制，避免卡顿。
4. 提供调试命令，能够输出某个 Steve 当前观察摘要。
5. 不要改变原有 Action 行为。

### 验收

- 可以查看 Steve 的观察结果。
- 危险等级能根据怪物、血量、夜晚等因素变化。
- 扫描不会明显卡顿。
- 通过 `mvn clean package` 构建。

## 6. 阶段 3：新增自主目标生成与选择

### 目标

让 Steve 在没有玩家命令时，可以根据环境自己产生候选目标。

### 任务

1. 实现候选目标生成机制。
2. 目标类型至少覆盖：
   - 生存
   - 探索
   - 好奇
   - 资源
   - 成长
   - 记忆
   - 社交
   - 社群
3. 目标生成规则：
   - 血量低时优先生存。
   - 饥饿低时优先找食物。
   - 夜晚且危险时优先找安全地点。
   - 缺少基础资源时优先采集。
   - 缺少基础工具时优先发展。
   - 发现新方块/新生物时产生好奇目标。
   - 安全且空闲时探索未知区域。
4. 实现目标选择机制：
   - 优先级高的目标优先。
   - 生存目标高于探索目标。
   - 避免短时间重复同一目标。
   - 最近失败过的目标降低优先级。
5. 提供调试命令，查看当前候选目标和被选目标。

### 验收

- Steve 在不同状态下能生成不同目标。
- 低血量/危险时优先生存。
- 安全时默认探索。
- 可以查看候选目标。
- 通过 `mvn clean package` 构建。

### Commit

`feat: 新增 MineMind 自主目标生成与选择机制`

## 7. 阶段 4：实现规则型自主循环

### 目标

让 Steve 在无玩家命令时可以周期性自主行动。

### 任务

1. 在 Steve 实体生命周期中接入自主循环。
2. 自主循环应遵循：
   - 如果自主模式未开启，不执行。
   - 如果正在执行玩家明确任务，不抢占。
   - 如果已有 Action 正在执行，继续执行。
   - 如果空闲且到达思考间隔，进行一次自主决策。
3. 自主决策流程：
   - 观察世界
   - 记录观察
   - 生成候选目标
   - 选择目标
   - 使用规则规划器转换为现有 Action/Task
   - 提交给现有 Action 执行系统
   - 记录结果
4. 先不要调用 LLM。
5. 如果目标无法转换为现有动作，必须安全降级，例如等待、探索或记录。
6. 所有行为必须 tick-based。

### 验收

- 开启自主模式后，不输入命令，Steve 会自己行动。
- 玩家下达明确任务时，自主模式暂停。
- 玩家任务结束后，自主模式恢复。
- 不会阻塞游戏。
- 通过 `mvn clean package` 构建。

### Commit

`feat: 实现 MineMind 规则型自主循环`

## 8. 阶段 5：新增聊天框引导与对话

### 目标

让玩家可以通过 Minecraft 聊天框自然语言引导 Steve，而不是只能用 GUI 或命令。

### 任务

1. 监听玩家聊天。
2. 当聊天内容提到智能体名字或配置触发词时，进入 MineMind 聊天处理。
3. 支持识别以下意图：
   - 询问状态
   - 询问原因
   - 询问下一步计划
   - 引导目标
   - 引导方向
   - 设置偏好
   - 标记地点
   - 警告危险
   - 停止当前动作
   - 恢复自主探索
   - 明确任务命令
   - 普通对话
4. 常见意图优先使用规则分类。
5. 不要每条聊天都调用 LLM。
6. 不确定时可以简短澄清。
7. 聊天回复要有冷却，避免刷屏。
8. 明确任务命令可以复用 Steve 原有 TaskPlanner。
9. 引导类输入要写入记忆，影响后续自主目标。

### 验收示例

玩家输入：

- `Bob，你在干什么？`
- `Bob，你为什么往北走？`
- `Bob，接下来准备干什么？`
- `Bob，记住这里是基地。`
- `Bob，别去东边。`
- `Bob，停止。`
- `Bob，继续自主探索。`

Steve 应能合理响应并改变状态或记忆。

### Commit

`feat: 新增 MineMind 聊天框引导与对话能力`

## 9. 阶段 6：新增玩家引导记忆

### 目标

让玩家通过聊天给出的引导成为长期上下文，而不是一次性回复。

### 任务

1. 记录玩家标记的位置：
   - 基地
   - 危险点
   - 资源点
   - 避难点
   - 村庄
   - 洞穴
2. 记录玩家偏好：
   - 不攻击动物
   - 夜晚不要外出
   - 优先找村庄
   - 优先建基地
   - 避免某个方向
   - 优先某个生物群系
3. 记录玩家警告：
   - 某个方向危险
   - 某片区域危险
   - 某类实体危险
4. 记录临时目标：
   - 今天优先做什么
   - 暂时不要做什么
5. 将这些引导接入目标生成与选择。
6. 玩家引导优先级低于生存，高于普通探索。
7. 引导记忆要持久化。

### 验收

- 玩家说“记住这里是基地”后，重启仍能记住。
- 玩家说“别去东边”后，东边探索优先级降低。
- 玩家说“优先找村庄”后，相关目标优先级提升。
- 通过 `mvn clean package` 构建。

## 12. 阶段 9：接入 DeepSeek

### 目标

在现有 LLM Provider 基础上增加 DeepSeek 支持。

### 任务

1. 阅读现有 LLM Provider 设计。
2. 以最小侵入方式增加 DeepSeek。
3. 支持配置：
   - API Key
   - Base URL
   - Model

   - Max Tokens
4. DeepSeek 可用于：
   - 原 Steve 命令模式
   - MineMind 自主规划
   - 后续社群规划
5. 不要硬编码 API Key。
